在上一篇文章[《Context is All You Need?》](/posts/context-is-all-you-need/)中，我从 Chat-bot 到 Agent 的演进出发，论证了"上下文"是理解当前 AI 架构演化的核心视角。文章结尾提出了两条扩容路径：纵向的多层缓存，和横向的多智能体。

纵向路径如今已相当成熟——从渐进式摘要到 MemGPT，从 RAG 到各种上下文压缩策略，业界在"如何在有限窗口内塞进更多有效信息"这个问题上做了大量扎实的工程和研究工作。

但横向路径——多智能体——的讨论往往停留在"并行加速"或"角色扮演"的直觉层面。我当时也是这么理解的：多智能体不过是 MapReduce 的翻版，通过并行探索来扩大搜索范围。

现在我觉得这个理解太浅了。多智能体真正重要的不是并行性，而是**上下文的分治**——通过将上下文分片，让每个智能体的有限窗口内装满与其任务高度相关的信息，从而最大化信息密度。

这篇文章试图从信息论和函数逼近论两个角度，给出一个更严格的论证。

---

## 一、概念澄清：多智能体 ≠ 子智能体

在开始之前，有必要做一个概念区分。

本文讨论的"多智能体"，不是目前常见的 sub-agent 模式。Sub-agent 本质上是一个工具调用——只不过被调用的"工具"恰好是另一个 agent。它可能在后台异步执行（如 fan-out/fan-in 架构），也可能是同步的（如 Cursor 的 sub-agent）。但无论哪种情况，sub-agent 都没有自己持久的上下文和专业积累——它被创建、执行、返回结果、销毁。

本文讨论的是具有**持久上下文**和**领域专长**的独立智能体之间的协作。它们各自维护自己的知识积累和记忆，在各自的专业领域深耕，通过某种通信机制协调工作。

这个区别很重要，因为它直接关系到一个根本问题：为什么需要多个独立的上下文，而不是一个足够大的上下文？

---

## 二、分工的本质：上下文分片

借鉴人类模式来设计 AI 系统并不总是合理的，但人类社会分工的逻辑可以帮助我们建立直觉。

理论上，一个人只要有无限的时间，完全可以串行完成一家大企业的所有工作。所以社会分工最直接的动机是**并行效率优化**。但对于 AI 来说，模型的推理速度远超人类，并行优化的收益并不像人类社会那样显著。如果仅仅为了并行，sub-agent 就够了。

但如果真的尝试过独自创业，你会发现：在一个从未涉足的领域（比如 marketing 对于工程师来说），你连该问什么问题都不知道。而一个专业的 CMO 可能一下午就能解决你困扰几个月的问题。

区别在哪？CMO 深耕 marketing 多年，积累了**深厚的领域上下文**。这些经验不是泛泛的"通用知识"，而是高度浓缩的、与特定领域高度相关的信息集合。

回到 AI 系统，道理是一样的：

> 多智能体的本质是**上下文的分片与分区**：在每个 agent 有限的上下文窗口中，尽可能装满与其任务高度相关的信息（最大化互信息），同时排除不相关的噪声。

这不是"让 AI 扮演人类角色"那么简单。目前有大量的多智能体工作在做角色扮演——AI 小镇有镇长和村民，AI 软件公司有 CEO、CTO、开发和测试，甚至有人把唐朝的[三省六部制做成了多智能体框架](https://github.com/cft0808/edict)。但如果不理解"分工"的信息论本质，这些设计就只是形式上的模仿。

## 三、信息论视角

### 3.1 注意力稀释：不只是比例问题

Transformer 的注意力机制用 softmax 给上下文中的每个 token 分配一个权重 $\alpha_i$：

$$
\alpha_i = \frac{\exp(q \cdot k_i / \sqrt{d})}{\sum_{j=1}^{n} \exp(q \cdot k_j / \sqrt{d})}
$$

直觉上可能会觉得：softmax 不就是一个比例吗？就算上下文变长、分母变大，最相关的 token 的权重还是最大的，模型照样能找到它，有什么问题？

问题在于注意力的输出不是 argmax，而是**加权求和**：

$$
\text{output} = \sum_{i=1}^{n} \alpha_i \cdot v_i
$$

每个 token 不管权重多小，都会通过自己的 value 向量 $v_i$ 贡献到最终输出中。当上下文里塞了大量弱相关的 token 时，它们每个分到一点点权重，但**加在一起的噪声可以盖过少量相关 token 的信号**。这不是"最大的还是不是最大的"的问题，而是"信号被噪声淹没"的问题。

更具体地说：假设上下文中有 $k$ 个相关 token 和 $n - k$ 个不相关 token。当 $n \gg k$ 时，即使每个不相关 token 的 $\alpha_j$ 很小，它们的总贡献 $\sum_{j \in \text{noise}} \alpha_j \cdot v_j$ 的量级可以和 $\sum_{i \in \text{signal}} \alpha_i \cdot v_i$ 相当——信噪比随 $n$ 增长而恶化。

多智能体分片等价于对每个 agent 缩小 $n$，同时保持 $k$ 不变——直接提升了加权求和中信号的占比。

Liu et al. (2023) 的"Lost in the Middle"[1] 就是一个很好的例子——在多文档问答任务中，关键信息放在中间位置时，模型的检索准确率显著下降。上下文越长，中间位置的信息越容易被"遗忘"。

### 3.2 不相关上下文的干扰：条件熵视角

假设任务 $Y$ 依赖于上下文中的相关子集 $X_R$ 和不相关子集 $X_N$。理想情况下，如果 $X_N$ 是纯噪声且与 $Y$ 完全独立，那么：

$$
H(Y \mid X_R, X_N) = H(Y \mid X_R)
$$

不相关信息不应影响任务的不确定性。但在实践中，模型并非完美的信息处理器——它可能错误地利用 $X_N$ 中的**虚假相关**（spurious correlation），导致：

$$
H(Y \mid X_R, X_N) > H(Y \mid X_R)
$$

也就是说，加入不相关信息反而**增加了**任务的不确定性。分片去噪等价于让每个 agent 只看到 $X_R$，从根源上规避了这个问题。

Shi et al. (2023)[2] 做过一个很直接的实验：在 GSM8K 数学题中插入不相关信息，模型准确率最多掉了约 20 个百分点。模型确实会被无关上下文带偏。

## 四、函数逼近视角：上下文就是参数

换一个角度。如果说信息论解释了"噪声为什么有害"，函数逼近论可以解释"为什么拆成多个简单的比一个复杂的好"。

先说 MoE 原始理论里的结论。Jordan & Jacobs (1994)[3] 在论文开头就直接点明了 divide-and-conquer 的统计代价：

> "Dividing the data can have favorable consequences for the **bias** of an estimator, but it generally increases the **variance**."

拆开不是免费的。但他们论证了：当目标函数是**局部简单、全局复杂**的（绝大多数真实任务都是这样），用多个简单局部模型拼起来，比用一个复杂全局模型效果更好。道理和数值分析一样——用很多小线段拼接（分段线性逼近）比用一个高次多项式拟合曲线更稳定。Wang & E (2025)[6] 给出了更严格的结果：浅层 MoE 可以高效逼近低维流形上的函数，绕过维度灾难。

这和多智能体有什么关系？关系很直接，但要先看到一个等价性：

**Agent 的上下文本质上就是模型的可变参数——只不过是在自然语言的离散空间中，通过前向传播而非梯度下降来更新的。**

传统神经网络里，参数通过反向传播更新，编码学到的知识。Agent 的上下文通过追加观察、工具返回、摘要来更新，编码的是工作记忆和领域知识。两者的功能角色完全一样：**决定模型面对同一个输入时的行为**。这不只是类比——Dai et al. (2023) 在 "Why Can GPT Learn In-Context?" 中证明了 Transformer 的 in-context learning 在特定条件下数学上等价于隐式梯度下降[7]。

一旦接受"上下文 = 参数"这个等价性，MoE 的 bias-variance 理论就直接适用了：

- **一个全局上下文**（把所有信息塞进一个窗口）= 一个全局模型。面临 bias-variance 困境：信息太多、太杂，模型要么抓不住重点（高偏差），要么被虚假相关带偏（高方差）。
- **多个分片上下文**（每个 agent 只持有与其任务相关的子集）= 多个局部模型。每个局部问题更简单，每个 agent 的上下文更聚焦，"拟合"质量更高。

而 MoE 理论中的**软分区**——一个数据点可以同时影响多个 expert，避免硬切割导致的方差爆炸——在多智能体系统中对应的就是**共享摘要层**：agent 之间不是完全隔离的，它们通过公共的高层摘要维持对全局的模糊认知，相当于分区边界是软的。

## 五、更强的模型会替代多智能体吗？

这个问题很自然。已经有不少研究在正面回答这个问题：

- Xu et al. (2026) 的 [Rethinking the Value of Multi-Agent Workflow](https://arxiv.org/abs/2601.12307) 在 7 个 benchmark 上测试发现，一个强单 agent 通过多轮对话就能追平同质多 agent 工作流的表现，还能通过 KV cache 复用获得效率优势。
- Gao et al. (2025) 的 [Single-agent or Multi-agent Systems? Why Not Both?](https://arxiv.org/abs/2505.18286) 指出，随着 OpenAI-o3、Gemini-2.5-Pro 这类前沿模型在长上下文推理和工具使用上的进步，多 agent 相对于单 agent 的优势在缩小。

那单个模型足够强的话，是不是就不需要多智能体了？

我的观点是：模型变强意味着它能**端到端地**解决更宏观的问题——比如从"写一个函数"进化到"从零搭建一个完整的服务"。但更宏观的问题也有更宏观的分工。一个能独立完成整个后端开发的 agent，仍然无法同时深耕前端设计、市场策略和法务合规。

**技术总是追着需求跑的。** 模型能力的提升扩大了单个 agent 的能力边界，但同时也提升了人们对 AI 系统的期望。当你期望 AI 不仅写代码还要理解业务、做决策、与外部系统协调时，单个上下文窗口的信息密度问题会在更高的层次上重新出现。

分工的颗粒度会随着模型能力的提升而变粗，但分工本身不会消失。

## 六、多智能体的代价：分布式系统的老问题

好处说了不少，但代价呢？如果代价管不住，前面说的都是白搭。

一旦采用多智能体架构，你实质上就是在构建一个**分布式系统**。而分布式系统有一个经典的不可能三角——**CAP 定理**：

- **C（一致性, Consistency）**：所有 agent 在同一时刻看到相同的世界状态
- **A（可用性, Availability）**：每个 agent 在任何时刻都能独立工作
- **P（分区容忍, Partition Tolerance）**：系统在 agent 之间通信中断时仍能运作

问题是：**P 是多智能体的前提条件。** 上下文分片本身就意味着分区——每个 agent 只持有世界状态的一个子集。所以我们只能在 C 和 A 之间做取舍：

- **CP 系统**（牺牲可用性换一致性）：严格的工作流编排，agent 之间有同步点。比如 MetaGPT[4] 的 PM → Architect → Engineer 串行流水线。一致性高，但存在阻塞，灵活性差。
- **AP 系统**（牺牲一致性换可用性）：agent 并行工作，最终汇总。比如多 agent 辩论、投票。并行度高，但各 agent 可能基于不一致的认知做决策。

我倾向于 AP 方向。原因是：我相信 agent 不等于人类，任何有效的协作机制都应该基于某种优化框架让 AI 自己**涌现**出来，而不是把瀑布流工作模式硬编码进系统。那些严格模仿人类组织架构的 CP 系统，本质上是在用人类的认知局限去约束一个不受同样局限的系统。

那么 AP 系统的核心挑战，就落在了**通信**上。

## 七、为什么"AI 版社交平台"走错了方向

目前市面上出现了一些"给 agent 用的社交平台"——模仿人类的即时通讯工具和社交网络，让 agent 之间发消息、建群组、发动态。

我觉得这个方向从根上就搞错了。

人类的沟通方式充满了**无奈的妥协**：

- **无法直接读取对方的思维**，只能打断对方手头的工作去对话，被询问者被迫切换上下文
- **拉群有信任顾虑**，人类的动机不透明，记忆和认知无法共享
- **点对点私聊**受限于隐私需求，也有人不愿意切换自己的上下文去参与群聊

但 agent 完全不受这些约束：

- 所有 agent 都会将上下文持久化到文件中，每个 session 的输出**完全由其上下文决定**
- **直接读取一个 session 的上下文**，和"向它发消息询问"在信息量上是等价的——而且前者不会污染对方的上下文
- Agent 没有"不愿意切换上下文"的情感成本

说白了，人类需要社交平台是因为我们的认知是封闭的、不透明的、有情感成本的。Agent 的认知是序列化的、可读的、没有情感负担的。把人类的沟通妥协照搬到 agent 世界，是在解决一个**不存在的问题**。

而且 agent 之间"聊天"本身就在**消耗 token**，降低上下文中的有用信息密度——这恰恰和上下文分片的初衷是矛盾的。

AP 系统真正需要解决的是通信复杂度问题。全连接通信的开销是 $O(n^2)$（$n$ 为 agent 数量），显然不可扩展。解决思路不是模仿人类的社交网络，而是从分布式系统的角度去设计通信架构。

## 八、那该怎么做

铺垫了这么多，该说说我的想法了。

**通信架构**：既然全连接不可行，一个自然的选择是**黑板模型**（Blackboard Architecture）——所有 agent 的上下文被集中索引和摘要，按照**渐进式披露**（progressive disclosure）的方式组织起来，从高层概要到底层细节，按需展开。不同的 agent 由于关注点不同，在渐进式披露的查询路径上会自然分化，从而实现公共记忆的**软性智能分区**。

**分工的度量**：前文论证了分工的本质是上下文与任务之间的信息密度优化。那么在实际操作中，如何决定一个新任务应该由哪个 agent 来处理？

最直接的思路是比较每个 session 的内容摘要与新任务之间的**语义相关性**，判断应该在现有 session 上继续（resume）、分叉（fork），还是创建新的 session。这本质上是一个路由问题。

为什么不直接用 embedding 模型做向量相似度比较？两个原因：

1. **端到端优先**。让 LLM 直接对任务描述和 session 摘要进行推理比较，比依赖中间的向量表示更可靠。
2. **向量相似度的脆弱性**。它能否准确表征 session 与任务之间的互信息，严重依赖摘要的生成质量和 embedding 模型的语义覆盖。这个链条太脆弱，不够通用。

更合理的方式可能是按照渐进式披露的层次结构，用 LLM 逐层比较和推理，在每一层做出路由决策——先在高层概要中筛选候选 session，再深入细节确认匹配度。

---

## 结语

从上一篇文章的"上下文扩容"讨论延续到这里，总结下来就是：

1. 多智能体的本质不是并行加速，而是**上下文的分治**——通过分片提高每个 agent 的信息密度
2. 信息论提供了理论支撑：注意力稀释和不相关上下文的干扰，都说明"大而全"不如"小而精"
3. 更强的模型不会消除分工的需求，只会改变分工的颗粒度
4. 多智能体本质上是分布式系统，受 CAP 约束，AP 方向更有前景
5. "AI 社交平台"模仿了人类沟通中的妥协，而非解决 agent 真正的通信问题
6. 黑板模型配合语义路由，可能是更合理的方向

多智能体不是万能药，但它回答了上一篇文章留下的一个问题：**当一个上下文窗口装不下整个世界时，该怎么办？**

答案不是拼命往一个窗口里塞更多东西，而是承认分治的必要性，然后把精力放在如何让分治后的系统高效协作上。
